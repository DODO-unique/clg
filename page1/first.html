BROKEN ACCESS CONTROL

Broken Access Control occurs when an application fails to enforce proper restrictions on authenticated users, allowing unauthorized access to sensitive data or functions.

Attackers can exploit these weaknesses to gain unauthorized access, potentially compromising entire systems, stealing sensitive data, or abusing admin privileges.

TECHNICALS

Types of Weak Access Controls

Insecure Direct Object References (IDOR)

Attackers manipulate URL parameters or references to access objects (e.g., files, database keys) belonging to other users by changing identifiers without appropriate access checks.

Missing Function Level Access Control

Access controls enforced only at the user interface level can be bypassed by directly calling server functions or APIs, allowing unauthorized operations such as administrative actions.

Overly Permissive Access Controls

When systems grant excessive permissions that violate the Principle of Least Privilege (PoLP), attackers can access higher-level functions or data they shouldn't have access to.

Failure to Invalidate Sessions

If session tokens remain valid after logout or session expiration, attackers can hijack sessions to impersonate authenticated users and access restricted areas.

Insecure Object References in APIs

Poorly secured API endpoints can expose confidential data or allow unauthorized actions if user authentication and authorization are not adequately enforced.

Eploitation Techniques

Privilege Escalation

Attackers exploit access control weaknesses to elevate privileges either horizontally (accessing another user's same-level privileges) or vertically (gaining higher-level privileges, e.g., admin).

Parameter Manipulation

Modifying URL parameters, form fields, or tokens to bypass access controls and access unauthorized data or functionalities.

Exploiting Unprotected Endpoints

Attackers discover and exploit API or service endpoints that lack proper access control, allowing unauthorized actions.

Misconfigurations

Incorrect access control settings or exposure of sensitive information in public URLs can lead to unauthorized data access.

Vulnerability Report - Broken Access Control

Issue #791

Opened by nmmorette on Nov 5, 2024

1. Vulnerability Overview

A critical vulnerability was identified in the web application WeGIA. This vulnerability allows an attacker to change the password of the admin user by sending a POST request to the control.php endpoint without requiring authentication or authorization. The lack of proper access controls and authentication checks on this endpoint poses a high risk of privilege escalation and potential unauthorized control over admin accounts.

2. Technical Details

Vulnerable Endpoint: POST /controle/control.php

During the test, the cookie was removed, and the field referring to the old password was also omitted.

3. Proof of Concept (POC)

Using curl:


curl -X POST https://demo.wegia.org/controle/control.php \
-H "Content-Type: application/x-www-form-urlencoded" \
-H "Origin: https://demo.wegia.org" \
-d "new_password=NewPassword123&confirm_password=NewPassword123"

This request demonstrates how an attacker can change the admin password without proper authentication.

For more details, view the full issue on GitHub: Vulnerability Report #791
